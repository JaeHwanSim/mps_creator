# AI - '26 3월 월간 MPS (내 MPS)

> **2026년 3월 내 MPS — Goose 기반 AI Agent 개발 시스템 + JIRA 기반 Gate Process 운영 기준선 확보**
>
> **입력 자료:** `MPS/references/3월/SD-AI` 센터 MPS 참고

## Mission

2월까지 구축한 JIRA 기반 Gate Process와 Goose 기반 AI Agent 시스템을 실제 개발에 적용하며, 운영 기준선과 추적성을 확보한다.

## 개요

Gate 프로세스는 단계별 불확실성을 통제하기 위해 각 Gate를 문서·데이터 기반 Go/Kill/Hold/Recycle로 운영한다. 3월은 2월에 만든 시스템을 "완성"하기보다 실사용에 적용하며 병목과 결함을 빠르게 제거해 견고한 기준선을 만드는 달이다.

## 목적

1. JIRA에서 Gate 운영(산출물·승인·Decision Log)이 실제로 돌아가도록 적용하고 추적성을 확보한다.
2. Goose 기반 AI Agent 시스템을 실제 개발에 적용해 협업 리드타임을 단축하고 품질 게이트를 강화한다.
3. "개발 완료와 동시에 의료기기 인허가를 투입할 수 있는 준비 상태" 최소 요건을 고정한다.

## 수요자 요구 사항

- **R&D 개발 조직:** 개발 완료 시점에 인허가 투입이 가능한 준비 상태(산출물·추적성·형상·위험관리 정합)
- **QA/품질:** Gate 산출물·승인·Decision Log가 시스템에 누적되는 추적성

## 현재 상태

### As-Is

- JIRA 기반 Gate Process가 미완성 상태로, Gate 산출물·승인·Decision Log가 일관되게 축적되지 않음.
- Goose 기반 AI Agent 시스템은 구성되어 있으나, 실제 개발 사이클에서 반복 적용 경험과 운영 규칙이 부족함.
- Gate 프로세스의 "문서와 데이터" 중심 원칙은 있으나, 실사용 연결(이슈–문서–코드)이 느슨함.

### To-Be

- JIRA에서 Gate별 필수 산출물 링크·승인 상태·Decision Log가 누락 없이 관리되어 "Gate 운영이 시스템으로 굴러감".
- Goose 기반 AI Agent가 개발 흐름(이슈→코드→테스트→PR→문서)에서 반복 사용되며 품질/보안 기준을 보조함.
- 산출물·이슈·코드가 추적 ID로 연결되어 DHF 관점 추적성이 확보되고 EA/ER 진입이 빨라짐.

## 기대하는 결과물

1. **JIRA 기반 Gate Process 운영 기준선** (워크플로우/이슈타입/필수필드/승인 규칙/Decision Log 템플릿)
2. **Gate 운영 대시보드** (Gate별 진행률·승인 대기·산출물 누락·리스크 현황)
3. **Goose 기반 AI Agent 운영 기준선** (Agent 역할/권한/프롬프트 패키지/운영 가이드, "제안=초안·승인=최종" 원칙)
4. **추적 ID 규칙** (PR 템플릿에 추적 ID 연결, 이슈–문서–코드 매핑)
5. **Agent 적용 개발 실적 리포트** (PR 4건 이상 적용 결과, 시간 절감/결함/리뷰 리드타임, 개선 백로그)

## Performance Objectives

### 성과 목표 1: JIRA 기반 Gate Process 운영 기준선 + Gate 운영 대시보드

> Gate의 "문서·데이터 기반 의사결정"을 JIRA에서 실제로 굴릴 수 있도록 운영 기준선을 확정하고 대시보드로 가시화한다.

**고정변수목표**
- JIRA Gate 워크플로우/이슈 체계: Gate 단위 이슈 타입(PA/EA/ER) 정의, 필수 필드(산출물 링크·승인자·Due date·상태) 표준화, Gate별 필수 산출물 체크리스트로 누락 차단, Decision Log 템플릿(결정/보류/추가확인/담당/기한) 내재화
  - 공략전략: Gate별 "필수 산출물"을 이슈 필드/체크리스트로 고정해 시스템적 누락 차단, Decision Log로 후속 지연 통제
- Gate 운영 대시보드: Gate 진행률·승인 대기·산출물 누락·리스크 Top N을 한 화면에서 확인, 주간 운영 리뷰(주 1회)에서 대시보드 기반 의사결정 루틴 고정
  - 공략전략: 대시보드를 단일 데이터 소스로 지정, 주간 회의에서 "대시보드 기반 의사결정만 허용"으로 운영 실효성 확보

**변동변수목표**
- Gate 운영 Pilot 리포트: 실제 개발 업무(최소 1개 기능/이슈 묶음)에 Gate 운영을 적용해 누락·병목·저항 포인트를 수집하고 개선 우선순위를 4월 백로그로 전환
  - 공략전략: 적용 범위를 최소 기능 단위로 고정, 주간 운영 리뷰에서 즉시 의사결정 후 백로그화

### 성과 목표 2: Goose 기반 AI Agent 운영 기준선 + Agent 적용 개발 실적

> Goose 기반 AI Agent를 "개발자가 매일 쓰는 도구"로 만들기 위해, 실제 개발에 적용하며 품질/보안 규칙을 포함한 운영 기준선을 확정한다.

**고정변수목표**
- Goose Agent 역할/권한/운영 가이드: Agent 역할 정의(이슈 정리·코드 스캐폴딩·테스트 생성·PR 리뷰·문서화), 권한/금지 규칙("제안=초안, 승인=최종", 민감정보 마스킹·금지 프롬프트), 개발자 협업 규칙(리뷰 체크리스트/DoD/테스트 필수)
  - 공략전략: 권한·금지 규칙을 명문화해 운영 가이드에 포함, "제안=초안" 원칙으로 결함 유입 차단
- Repo/CI/PR 표준 + 품질 게이트: PR 템플릿/리뷰 체크리스트/테스트 실행 규칙 표준화, 추적 ID 연결 규칙을 PR 템플릿에 고정, CI 품질 게이트(필수 테스트/정적 분석) 미통과 시 병합 불가
  - 공략전략: 추적 ID 규칙을 PR 템플릿에 강제해 이슈–문서–코드 연결, CI 게이트로 자동 품질 통제

**변동변수목표**
- Agent 적용 개발 실적 리포트: 실제 개발에서 PR 4건 이상에 Agent 적용(최소 2명 개발자 참여), "시간 절감/결함 감소/리뷰 리드타임/문서 생성 품질"을 정리하고 문제를 4월 개선 백로그로 전환
  - 공략전략: 핵심 개발 흐름에 Agent 적용 후 정량·정성 지표를 리포트화, 발견 문제를 4월 목표 입력으로 사용

## Risks

> 미션 전체 관점. 외부환경 1개 이상 + 내부역량 1개 이상.

- **외부환경(일정):** 개발 일정/우선순위 충돌로 Pilot 적용이 밀림
  - 대응방안: 적용 범위를 최소 기능 단위(1~2개)로 고정, 주간 운영 리뷰에서 즉시 의사결정
- **외부환경(규제):** 규제/품질 요구사항(추적성/DHF) 해석 불일치
  - 대응방안: Gate 프로세스 문서 기준으로 체크리스트화해 JIRA 필수 필드/산출물 체크에 반영
- **내부역량(운영 형식화):** JIRA 입력 부담으로 운영이 형식화됨
  - 대응방안: "필수 최소 필드"만 강제, 대시보드 기반 주간 운영으로 실효성 확보
- **내부역량(품질/보안):** Agent 환각·생성 코드 결함 유입, 코드/데이터 유출 우려
  - 대응방안: CI 품질 게이트 강제 + PR 리뷰 필수 + "제안=초안" 원칙, 권한 최소화·민감정보 마스킹·로그/감사 추적 강화
