# SW - '26 3월 월간 MPS (내 MPS)

> **2026년 3월 내 MPS — Smile Design 2.0 PA 산출물 세트 기준선 확보 + 기본 기능 Prototype(Alpha)으로 기술 리스크 조기 확인**
>
> **입력 자료:** `MPS/references/3월/SD-SW` 센터 MPS 참고

## Mission

Smile Design 2.0 PA 단계 산출물 세트(SDP·RMP·CMP·보안·유지보수·RA 등급)를 운영 가능한 기준선으로 확정하고, 기본 기능 Prototype/Alpha로 기술 리스크를 조기에 확인한다.

## 개요

2월 Kick-off에서 합의한 과제 목적/기간/Scope를 기준선으로 PA 단계 산출물을 완성한다. 동시에 EA 진입 전 리스크가 큰 구간을 빠르게 확인하도록 기본 기능을 MVP Path 중심으로 Prototype/Alpha 수준으로 구현한다. 이 과정에서 Gate 기반 개발 시스템(산출물·승인·형상·변경관리)이 실제로 동작하는지 검증한다.

## 목적

1. PA 단계 필수 산출물 세트를 운영 가능한 기준선으로 확정하여 착수 승인 체계를 안정화한다.
2. 기본 기능 Prototype/Alpha로 기술 리스크를 조기에 식별하고 과제 진행 가능성을 검증한다.
3. Gate 기반 개발 시스템(문서·데이터 기반 승인/검토/형상)을 실제로 운영해보며 실효성을 검증한다.

## 수요자 요구 사항

- **Stakeholder/경영:** Scope·일정·역할이 명확한 상태에서 통제 가능한 착수 승인 근거
- **QA/품질:** 초기 단계부터 형상·보안·유지보수 관점이 포함된 거버넌스 체계
- **개발팀:** PA Gate 통과 기준(산출물 체크리스트·승인 규칙) 명확화
- **제품/SAD 2.0 통합:** 초기 실행물(Prototype)로 방향성 확보

## 현재 상태

### As-Is

- 2월 Kick-off에서 과제 목적/기간/Scope 합의는 존재
- PA 단계 산출물(SDP/RMP/CMP 등)의 완성도·운영 방식은 실제 적용 미검증
- 기본 기능 구현은 착수 전, Gate 시스템(승인/변경/형상)은 문서만 존재

### To-Be

- PA 단계 필수 산출물 세트가 기준선으로 확정되고 승인/변경/형상 체계가 동작
- 기본 기능 Prototype/Alpha 초기 실행물 확보로 기술 리스크 조기 가시화 (재현성·입출력 스펙·평가 harness는 아직 미비 — 4월 과제)
- Gate 기반 개발 시스템 운영 검증 결과(개선점 포함)가 리포트로 정리됨

## 기대하는 결과물

1. **PA 단계 산출물 세트**(기준선): SDP(개발계획서), RMP(위험관리계획서), CMP(형상관리계획서), 보안 및 유지보수 계획서, RA 전략 및 제품 등급(Class) 정리
2. **기본 기능 초기 실행물**(Prototype/Alpha): "데이터 입력 → 핵심 처리 → 결과 확인" 최소 루프, MVP Path 중심 (산발적 실험 코드 수준, 입출력 스펙·재현성·평가 harness는 미비)
3. **Gate 기반 개발 시스템 운영 검증 리포트**(운영 결과·문제점·개선안·표준 템플릿/체크리스트)

## Performance Objectives

### 성과 목표 1: Smile Design 2.0 PA 산출물 세트 기준선 확보

> PA 단계 필수 산출물을 실제 개발 운영에 사용할 수 있는 수준으로 완성하고, 승인/변경/형상/보안 체계가 동작하도록 기준선을 확정한다.

**고정변수목표**
- PA 산출물 세트 작성·기준선 확정: SDP, RMP, CMP, 보안 및 유지보수 계획서, RA 전략 및 제품 등급(Class)
  - 공략전략: Kick-off 합의사항을 문서 기준선으로 반영, 변경관리 규칙(변경 요청/승인/기록) 정의, QA 참여 기준과 산출물 버전 관리 규칙 명확화
- 형상(CMP) 기준선 확보: 산출물·문서에 대한 형상 체계(Git repo 구조·체크인 규칙·버전 태깅) 적용
  - 공략전략: CMP 규칙을 산출물에 먼저 적용해 형상 토대를 다짐 (※ 파이프라인 코드에 형상 규칙 적용은 4월 과제)

**변동변수목표**
- Stakeholder 승인/검토 리드타임 단축 또는 안정화: 승인 지연 원인 분석 및 개선 액션 도출
  - 공략전략: Pre-Read 사전 공유 + Decision Log로 쟁점·결정사항을 문서화하여 리드타임 관리

### 성과 목표 2: 기본 기능 Prototype/Alpha 초기 실행물 (기술 리스크 조기 확인)

> EA 진입 전에 기술 리스크가 큰 구간을 조기 확인하도록 기본 기능을 MVP Path 중심으로 Prototype/Alpha 수준으로 구현한다.

**고정변수목표**
- 기본 기능 Prototype/Alpha 실행물: "데이터 입력 → 핵심 처리 → 결과 확인" 최소 루프 확보 (입출력 스펙 미정형, 시드/config/의존성 고정 미비 — 재현 가능 코드 기준선은 4월)
  - 공략전략: 최소 단위 실행 흐름(MVP Path) 중심으로 범위 제한, 산발적 실험 코드라도 핵심 처리가 돌아가는 최소 루프 우선 확보

**변동변수목표**
- Prototype에서 발견된 리스크/제약의 PA 리스크 관리 반영: RMP 보강 근거로 환류
  - 공략전략: 구현 과정의 리스크/제약을 문서화하여 RMP 위험 항목에 반영, 다음 달(4월) 재현 가능 기준선의 전제 조건 명시

### 성과 목표 3: Gate 기반 개발 시스템 운영 검증

> PA 산출물 작성/승인/변경관리와 Prototype 개발을 병행 운영하며, Gate 시스템이 조직·일정·개발 흐름에 적합한지 검증하고 개선안을 도출한다.

**고정변수목표**
- Gate 운영 검증 리포트: 운영 결과·문제점·개선안·표준 템플릿/체크리스트 정리
  - 공략전략: Gate 산출물 템플릿/체크리스트를 실제 흐름에 적용해 개선점 수집, Go/Kill/Hold/Recycle 의사결정 근거/증적 구조가 작동하는지 점검

**변동변수목표**
- 개발팀/QA/Stakeholder 병목(승인·리뷰·변경관리) 도출 및 운영 규칙 보정
  - 공략전략: 각 관점별 병목 항목을 식별하고 승인 단계 최적화·템플릿 단순화 개선안 도출

## Risks

> 미션 전체 관점. 외부환경 1개 이상 + 내부역량 1개 이상.

- **외부환경(Stakeholder 승인):** Stakeholder 관점 차이로 PA 산출물 승인 지연 가능
  - 대응방안: Pre-Read(사전 공유) + Decision Log로 쟁점/결정사항 문서화, 리드타임 관리
- **내부역량(리소스 분산):** PA 문서 작업과 Prototype 개발 병행으로 리소스 분산
  - 대응방안: "필수 산출물 우선" 원칙으로 범위 통제, 기본 기능은 MVP Path로 제한
- **내부역량(Gate 프로세스 과중):** Gate 프로세스가 과도하게 무거워져 개발 속도 저하
  - 대응방안: 운영 검증 리포트에서 템플릿 단순화·승인 단계 최적화 개선안 도출
