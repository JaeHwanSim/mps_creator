# MPS Creator 문서 구조 설계

> **날짜:** 2026-06-22
> **상태:** 설계 승인됨 → 구현 대기

## 1. 배경

이 저장소(`mps_creator`)는 R&D 센터의 **MPS(Multi-Period Planning)** — 년간/월간/주간 등 주기별로 "내가(우리 조직이) 해야 할 업무와 달성 방안"을 정리하는 문서 — 를 체계적으로 만들고, 궁극적으로 **자동 작성하는 agent 시스템**으로 발전시키기 위한 프로젝트다.

현재(`MPS/references/3월/`)에는 Device / SW / AI 세 조직의 **3월 월간 MPS**만 작성되어 있고, **4/5월이 비어 있다.**

- 세 MPS 문서는 **동일한 뼈대**를 따른다 (Mission → 개요 → 목적 → 수요자 요구사항 → 현재상태(As-Is/To-Be) → 기대 결과물 → Performance Objectives(고정변수목표/변동변수목표 + 공략전략) → Risks(외부/내부 + 대응방안)).
- 공통 맥락: Gate 기반 개발 프로세스(PA/EA/ER), 의료기기/의료소프트웨어, 5월 SIDEX 전시.

> **구조 정정(2026-06-23):** `MPS/references/`는 상위 **센터 MPS**(내 MPS 작성의 입력 자료)이고, **내 MPS**는 `MPS/created/`에 작성한다. 본 스펙의 MPS 문서 규격(§5)은 양쪽 모두에 동일하게 적용한다. (위 배경 설명의 "3월만 작성, 4/5월 비어있음"은 `created/`(내 MPS) 기준이며, `references/` 센터 MPS는 3·4월·연간이 별도로 존재한다.)

## 2. 목표

### 단기 (지금)
- 3월까지의 작성 패턴을 **문서 규격으로 고정**한다.
- **4/5월 월간 MPS**(Device/SW/AI 각각)를 사람이 Claude와 함께 수동으로 완성한다.
- 이를 통해 자동화에 필요한 충분한 샘플 데이터를 축적한다.

### 장기 (최종 골)
- MPS 플래닝을 **자동으로 작성하는 agent 시스템** 구축.

### 유연성 (미래 고려)
- 현 시점은 **조직명(Device/SW/AI) 베이스**로 MPS를 구성한다.
- 이후 달에는 **구체적인 프로젝트명 베이스**로 전환할 수 있다. (규격은 유지하되 분류 축만 교체 가능하도록 느슨하게 설계.)

## 3. 범위 (이번 설계의 대상)

이번 설계는 **문서 구조(README.md / CLAUDE.md / roadmap.md)를 만드는 것**에 한정한다.
4/5월 MPS 본문 작성 자체는 별도 후속 작업이다.

## 4. 설계 — 파일 역할 분담

| 파일 | 독자 | 역할 |
|------|------|------|
| `README.md` | 저장소를 처음 보는 사람 (미래의 나 포함) | 프로젝트 개요, 현재 상태, 목표, 디렉토리 구조. 가볍게, 디테일은 링크 |
| `CLAUDE.md` | Claude (매 세션 자동 로드) | 프로젝트 목적·Claude 역할·**MPS 문서 규격(뼈대+작성 규칙)**·진행 상황·다음 단계 |
| `docs/roadmap.md` | 단계별 로드맵 | Phase 1(수동) → Phase 2(데이터 축적) → Phase 3(자동화) |

**원칙:**
- `README.md`는 개요 수준. 디테일·규격·로드맵은 각각 `CLAUDE.md`, `docs/roadmap.md`로 링크.
- `CLAUDE.md`에 **MPS 규격**을 명시 → 4/5월 수동 작성과 향후 자동화가 모두 이 규격을 따름 (자동화의 기반).
- 진행 상황(어느 달까지 작성됨)은 `CLAUDE.md`의 한 섹션 + `roadmap.md`에서 관리. 매 업데이트 시 이 둘을 동기화.

## 5. MPS 문서 규격 (CLAUDE.md에 명시할 내용)

### 5.1 카디널리티 (개수 관계) — 핵심

| 요소 | 개수 |
|------|------|
| Performance Objectives | **2~3개** |
| ├─ 고정변수목표 (각 PO 내) | **1개 이상 (복수 가능)**, 각각 공략전략(1:1) |
| ├─ 변동변수목표 (각 PO 내) | **0개 이상 (복수 가능)**, 각각 공략전략(1:1) |
| Risks (최상위 섹션, 미션 전체) | **외부환경 1개 이상 + 내부역량 1개 이상**, 각각 대응방안 |

> 즉, Performance Objective 2~3개이며 각 PO마다 고정/변동목표는 여러 개 가능(각 1개당 공략전략 1개). Risk는 별도 최상위 `## Risks` 섹션에 미션 전체 관점으로 외부/내부 각각 1개 이상 작성.

### 5.2 문서 구조 (트리)

```
MPS 문서
├─ 제목:  # [조직] - 'YY M월 월간 MPS
├─ 요약 인용구 (>)
├─ Mission
├─ 개요
├─ 목적                      (번호 리스트)
├─ 수요자 요구 사항
├─ 현재 상태                 As-Is / To-Be
├─ 기대하는 결과물           (번호 리스트, 각 산출물에 버전 명시)
├─ Performance Objectives    (2~3개)
│   └─ 성과 목표 N
│       ├─ 고정변수목표 (1+) — 각각 공략전략 1개(1:1)
│       └─ 변동변수목표 (0+) — 각각 공략전략 1개(1:1)
└─ Risks                     (최상위, 미션 전체)
    ├─ 외부환경 (1+) + 대응방안
    └─ 내부역량 (1+) + 대응방안
```

### 5.3 작성 규칙

- 모든 산출물은 버전(v1.0 등)을 명시.
- "고정변수목표"는 반드시 달성, "변동변수목표"는 달성 시 가산.
- 각 변수목표(고정/변동) **1개당 공략전략 1개**를 1:1로 매핑.
- **Risk는 별도 최상위 `## Risks` 섹션에 미션 전체 관점으로 작성** (PO 내부에는 Risk를 두지 않음). 외부환경 최소 1개 + 내부역량 최소 1개, 각각 대응방안. (3월 문서의 최상위 Risks 표 형태와 일치.)
- 한국어로 작성. Gate 프로세스 용어(PA/EA/ER, Go/Kill/Hold/Recycle, Decision Log 등)는 기존 문서 표현을 유지.
